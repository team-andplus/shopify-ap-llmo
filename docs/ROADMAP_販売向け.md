## 販売するために「詰めたい」ところ（README ベース）

### 必須に近いもの

1. **llms.full.txt をどう扱うか**
   - README: 「アプリが自動生成」
   - 現状: リンクだけ。中身は 404 のままでもよい、とアプリホームに記載。
   - **選択肢**
     - **A**: いまは「ユーザーが用意する」と明記し、自動生成は「今後のバージョンで対応」とする（販売は可能）。
     - **B**: 販売前に「自動生成」を実装する（ストアの商品/ページなどを集約してテキスト生成する API 等）。

2. **docs/ai/〇〇.md（複数 md）**
   - 現状は `docs/ai/README.md` のみリンク。
   - **選択肢**
     - アプリで「追加するパス」を設定できるようにする（例: `docs/ai/README.md` + `docs/ai/faq.md` を登録）。
     - または「README のみ。他はストア側で適宜ページを作成」とドキュメントで明記する。

3. **アプリの「設定」の有無**
   - 現状: テーマブロックの on/off 以外にアプリ側の設定はなし。
   - 販売時にあるとよいものの例:
     - **URL のカスタム**（llms.txt のパスを `/llms.txt` 以外にする、など）
     - **リンクの on/off を個別に**（llms.txt だけ出す、など）
   - 必須かどうかは「無料配布のままか」「有料化するか」で判断。

4. **ストア審査・公開に必要なもの**
   - プライバシーポリシー・サポート先（アプリホームのフッターにリンクはあり）。
   - アプリ説明文・スクリーンショット（ストアリスティング用）。
   - スコープ・データ利用の明示（README の「思想」と整合）。

### あると差別化できるもの（README の「導入支援 10〜20 万」と合わせる）

5. **オンボーディング**
   - 初回インストール時「テーマカスタマイズで LLMO head を追加してください」を 1 画面で案内。

6. **動作確認の案内**
   - 「ストアフロントのソースで head 内の link を確認」を、アプリ内で簡潔に案内（いまのアプリホームの「動作確認」を少し整理するだけでも可）。

7. **llms.txt / docs/ai の「たたき」テンプレ**
   - ユーザーが自分で書く前提だが、「こんな文面でよい」例をアプリからコピペできるようにする。

8. **llms.txt 文章生成（インプット → プロンプト機械生成 / API）**
   - **複数インプットフィールド**でプロンプトを機械生成。**OpenAI API Key をセットしていれば**そのプロンプトで AI 生成し、採用で llms.txt 更新。**md ファイルの使用もフィールドで指定**。詳細は **docs/機能案_llms-txt_文章生成.md**。

9. **llms.full.txt 毎日自動生成**
   - **コレクションごとの商品・FAQ・ロケーション・ブランド・配送支払い**を毎日自動生成し、Files API で llms.full.txt を更新。詳細は **docs/機能案_llms-full-txt_毎日自動生成.md**。

## Shopify で「任意の .txt / .md を置けるか」

**結論: ストアのルート（例: store.com/llms.txt）に任意のファイルを置くことはできません。**

| 方法 | できること | 制約 |
|------|------------|------|
| **管理画面の「コンテンツ → ファイル」** | .txt / PDF などをアップロード可能 | 配信URLは **Shopify CDN**（例: `cdn.shopify.com/s/files/1/0xxx/xxxx/files/llms.txt`）。`store.com/llms.txt` のようなルートURLにはならない。 |
| **Pages（ページ）** | 本文をテキストで書ける | URL は **/pages/ハンドル**（例: `/pages/llms`）。カスタムテンプレで `Content-Type: text/plain` を出せば「テキストとして」配信は可能。 |
| **テーマの templates** | **robots.txt だけ**特別 | `robots.txt.liquid` だけが `store.com/robots.txt` で公開される。他の .txt をルートで配信する仕組みはない。 |
| **Files API（GraphQL）** | アプリからファイル作成・更新 | 作成したファイルは「コンテンツ → ファイル」に入り、**CDN URL** が発行される。同一URLを維持して中身だけ差し替える運用が可能。 |

→ **llms.txt / .md を「ユーザーが用意する」場合の現実的な選択肢**
- **A) コンテンツ → ファイル** にアップロードしてもらい、発行された **CDN URL** をアプリの設定で入力（またはアプリがファイルを作成してそのURLを保持）。
- **B) ページ** でハンドル `llms` などのページを作り、テキスト用テンプレで **/pages/llms** をテキスト配信にする（URLは `/llms.txt` ではない）。
- **C) 外部ホスティング**（自前サーバー・GitHub Pages 等）に置き、その **絶対URL** をアプリ設定で入力。

## 方針メモ（ llms.txt / llms.full.txt / .md ）

### llms.txt（思想・プロトコル）

- **中身**: ユーザー準備でよい。**「どういう意図で書くべきか」をアプリ側で示すことは必要**（ガイド・プロトコル説明）。
- **設置**: Shopify ではルートに .txt を置けないので、
  - アプリで「設定 → llms.txt の URL」を入力させる（Files の CDN URL または外部 URL）、
  - または **Files API でアプリが「ひな形」の .txt を生成**し、その CDN URL をブロックに渡す＋ユーザーは管理画面の Files から中身を編集、  
  のどちらか（または両方対応）が現実的。

### llms.full.txt（サイト内容の自動生成）

- **中身**: **サイトの内容をすべて自動生成する**。トリガーは **定時処理（cron）を主**、**手動ボタンは副**。
- **実装イメージ**:
  - cron で「ストアの商品・ページ等を取得 → 要約・一覧テキストを生成 → 1 ファイルとして保存」。
  - 保存先: **Files API でファイル作成 or 更新**し、**発行された CDN URL** をアプリの設定（DB 等）に保持。Theme Extension にはその URL を渡して `<link href="そのURL">` を出す。
  - ボタン: 管理画面に「今すぐ生成」ボタンを用意し、同じ生成ロジックを実行してから Files を更新。

### docs/ai/*.md（思想に応じて任意に設置）

- **中身**: 思想に応じてユーザーが任意に設置。
- **設置**: 同様に、**Shopify では任意の .md をルートに置けない**。
  - **コンテンツ → ファイル** に .md をアップロード → その **CDN URL** をアプリで登録、
  - または **外部 URL** を登録、
  - アプリは「登録された URL の一覧」を設定として持ち、Theme Extension に渡して **複数 `<link>` を出力**する形が現実的。
- **「思想に応じて任意に」** → アプリ側では「**追加するリンク（URL またはパス表記）を複数登録できる設定**」を用意するイメージ。

## 実装に必要なこと（この方針だと）

| 項目 | 必要なこと |
|------|------------|
| **llms.txt** | ① 意図・プロトコルのガイド文をアプリ内に用意 ② URL 設定 または Files API でひな形作成＋その URL をブロックへ渡す |
| **llms.full.txt** | ① 商品・コレクション・ページ等を取得する API ② テキスト生成ロジック ③ Files API でファイル作成/更新 ④ 発行 URL を保存 ⑤ cron のスケール（＋「今すぐ生成」ボタン） ⑥ Extension に「full の URL」を渡す仕組み（アプリ設定 or メタフィールド等） |
| **docs/ai/*.md** | ① アプリ設定で「追加リンク（URL）」を複数登録 ② Extension がそのリストを受け取り、複数 `<link>` を出力 ③ README.md は従来どおり 1 本固定 or 設定の 1 件目として扱う |
