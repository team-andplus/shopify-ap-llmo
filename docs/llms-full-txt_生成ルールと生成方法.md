# llms.full.txt の生成ルールと生成方法

実装時の参照用。既存の「機能案」「壁打ち」を整理し、**生成ルール**（何を・どう書くか）と**生成方法**（どこからデータを取るか・いつ動かすか）を一括で定義する。

---

## 1. 生成ルール（何を・どう書くか）

### 1.1 役割と読者

- **llms.full.txt** = サイトの**事実の一覧・要約**。思想（llms.txt）ではなく、**ストアの構造と中身**を LLM が解釈しやすい形で並べる。
- 読者: LLM・エージェント。人間が読んでも「このストアの全体像」が分かること。

### 1.2 含める内容（セクション）

| セクション | 内容 | 元データ |
|------------|------|----------|
| **ストア概要** | ストア名・1〜2 文の説明（あれば）。 | ストアメタ・llms.txt の title/roleSummary を参照してもよい（任意） |
| **コレクションと商品** | コレクション名ごとに、商品タイトル（または短い説明）の一覧。 | Admin API: Collections + Products |
| **ブランド・ベンダー** | 取り扱いブランドまたは vendor の一覧。 | 商品の vendor / メタフィールド |
| **ロケーション** | 店舗名・住所・営業時間の要約。 | Locations API |
| **配送・支払い・返品** | 送料の考え方、対応決済、返品ポリシーの要約。 | ポリシーページ / Shipping・Payment 設定 |
| **FAQ** | Q&A 形式。 | ストアの FAQ ページ・メタフィールド・またはアプリの FAQ 設定（あれば） |

※ FAQ はストアごとにデータ源が違うため、取得手段は別途検討（メタオブジェクト・Scrape・連携アプリ等）。

### 1.3 フォーマット・文体

- **プレーンテキスト**（または Markdown の見出し・リスト程度）。LLM がパースしやすいこと優先。
- **見出し**: `## セクション名` のような階層で区切る。
- **事実ベース**: 誇張・キャッチコピーは入れない。商品名・価格帯・条件など、事実だけ。
- **サイズ**: 50KB 以下を目安（llms.txt 周りの慣習）。商品数が多い場合は「コレクションあたり N 件」「直近 M 件」で打ち切り、または要約のみにする。

### 1.4 AI 補正（任意・有料で提供）

- **入力**: 上記ルールで作った「生データの羅列」テキスト。
- **プロンプト方針**: 「冗長を削り、事実ベースのまま、見出しとリストを維持した llms.full.txt 用の要約に整えて。誇張は入れない。○○字（または ○○ トークン）以内。」
- **出力**: 同じ構成で読みやすくした 1 本のテキスト。生データの羅列だけでも llms.full.txt としては成立する。AI 補正は「読みやすさ・長さの最適化」の付加価値。

## 2. 生成方法（どこから・いつ・どう保存するか）

### 2.1 データ取得

- **Shopify Admin API**（REST または GraphQL）で取得する想定。
  - コレクション一覧・商品一覧（コレクション別または全商品のうち N 件）
  - ロケーション（Locations）
  - ポリシー（Shipping / Payment など）は設定またはページ取得
  - ブランド・vendor: 商品の vendor を集約、またはメタオブジェクト
- **Storefront API** でも可（公開情報のみなら Storefront の方が一貫する場合あり）。実装時にどちらが楽かで選択。
- 取得結果を **見出し＋リスト形式のプレーンテキスト** に整形する（この時点で「生 llms.full.txt ドラフト」）。

### 2.2 処理フロー

1. **データ取得** → ストアごとにコレクション・商品・ロケーション・ポリシー等を取得し、1 本のテキストに整形。
2. **（任意）AI 補正** → ユーザーが OpenAI API Key を設定していれば、そのテキストを 1 回 API に渡し、要約・整形。未設定なら生データのまま。
3. **保存** → 結果を **Shopify Files API** で `llms.full.txt` としてアップロード。既存ファイルがあれば中身を更新（fileUpdate）。初回は stagedUploadsCreate → fileCreate。返ってきた URL を **DB（LlmoSettings 等）に保存** し、App Proxy はその URL に 302 する。

### 2.3 実行タイミング

- **毎日 1 回**: 決まった時刻にバッチで「有料プランのストア」に対して上記フローを実行。cron・スケジュールジョブ・Trigger.dev 等でトリガー。
- **手動「今すぐ生成」**: 管理画面からボタンで実行してもよい（有料プランのみ。または無料は 1 回だけなど制限を検討）。

### 2.4 技術メモ

- **DB**: llms.txt と同様、`LlmoSettings` に `llmsFullTxtFileUrl`（または類似カラム）を追加し、Files の URL を保持。App Proxy の `llms.full.txt` はこの URL へリダイレクト。
- **サイズ・打ち切り**: 商品・コレクションが非常に多い場合は「コレクションあたり最大 K 件」「全体で最大 N 商品」などで打ち切り、末尾に「（他に ○○ 件）」と書くか、要約のみにする。
- **トークン（AI 補正時）**: 入力が長いので、モデルと max_tokens を指定し、プロンプトで「要約して ○○ 字以内」を明示。段階要約（ first pass でセクションごと要約 → 結合）も検討可。

## 3. まとめ（実装チェック用）

| 項目 | 内容 |
|------|------|
| **生成ルール** | ストア概要・コレクション/商品・ブランド・ロケーション・配送支払い・FAQ を、見出し＋リストのプレーンテキストで。事実ベース・50KB 目安。AI 補正は任意で要約・整形。 |
| **データ取得** | Admin API（または Storefront API）でコレクション・商品・ロケーション・ポリシー等を取得。 |
| **生成方法** | 取得 → 整形 → （任意）OpenAI で補正 → Files API でアップロード → DB に URL 保存。 |
| **実行** | 毎日 1 回バッチ ＋ 手動「今すぐ生成」（有料プラン想定）。 |
