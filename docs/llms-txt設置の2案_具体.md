# llms.txt の設置：2 案の具体イメージ

「URL を設定で入力」と「アプリがひな形を作成」の違いを、**ユーザーが何をするか**と**システムが何をするか**で整理する。

**採用方針**: **案B**（アプリが Files API でひな形の llms.txt を作成し、その URL をブロックに渡す）。

---

## 前提：head に出すのは「URL」だけ

Theme Extension のブロックでは、**&lt;link&gt; の href に「どこか 1 つの URL」を入れる**だけ。

- 今: `{{ shop.url }}/llms.txt` → ストアルートの `/llms.txt`（Shopify では存在させられないので 404）
- これから: **「llms.txt の実体がある URL」** を 1 本、ブロックに渡す必要がある。

その「1 本の URL」をどう用意するかが、次の 2 案。

---

## 案 A：ユーザーが「llms.txt の URL」をアプリ設定で入力する

### ユーザー視点の流れ

1. **llms.txt の実体をどこかに用意する**
   - **パターン A-1（Shopify の Files）**  
     - 管理画面 **コンテンツ → ファイル** で .txt をアップロード。  
     - 表示される「ファイルの URL」をコピー（例: `https://cdn.shopify.com/s/files/1/0123/4567/files/llms.txt?v=...`）。
   - **パターン A-2（外部）**  
     - 自前サーバーや GitHub などに llms.txt を置き、その URL を用意（例: `https://example.com/llms.txt`）。

2. **アプリを開き「設定」画面へ**
   - 例: 「AP LLMO」アプリ → 「設定」または「llms.txt の設定」。

3. **「llms.txt の URL」欄に、上でコピーした URL を貼って保存**
   - アプリはその文字列を保存するだけ（DB やメタフィールドなど）。

4. **テーマで「LLMO head」ブロックを有効にする**
   - ブロックは「アプリが渡した llms.txt 用 URL」を参照する（下記の技術側でどう渡すか）。

5. **ストアの &lt;head&gt; には、その URL が link で出る**
   - 例: `<link rel="alternate" type="text/plain" href="https://cdn.shopify.com/.../llms.txt" ... />`

### 技術側のやること（イメージ）

| 役割 | やること |
|------|----------|
| **アプリ（管理画面）** | 設定画面で「llms.txt の URL」入力欄を 1 つ用意。保存時に **アプリ所有のメタフィールド**（例: `namespace.key` = `llmo.llms_txt_url`）にその URL を書き込む。 |
| **Theme Extension（Liquid）** | ブロック内で `app.metafields.llmo.llms_txt_url` を参照。値があれば `<link href="{{ app.metafields.llmo.llms_txt_url }}" ... />`、なければ出さない（または従来の `shop.url` の説明用リンクのまま）。 |
| **データの流れ** | ユーザー入力 URL → アプリがメタフィールドに保存 → ストアフロント描画時に Liquid がその値を読む → head に 1 本の link が出力される。 |

**イメージ図（A 案）**

```
[ユーザー] ファイルをアップロード or 外部に配置
       ↓
[ユーザー] アプリの「llms.txt の URL」にそのURLを貼って保存
       ↓
[アプリ] メタフィールド（llmo.llms_txt_url）に URL を保存
       ↓
[テーマブロック] app.metafields.llmo.llms_txt_url を読む
       ↓
[head] <link href="(そのURL)" ... />
```

- **メリット**: 実装が単純。既にファイルがある場合や外部 URL を使いたい場合にそのまま使える。
- **デメリット**: 「コンテンツ→ファイル」の操作と「URL をコピーして貼る」をユーザーが自分でやる必要がある。

---

## 案 B：アプリが Files API で「ひな形」の llms.txt を作成し、その URL をブロックに渡す

### ユーザー視点の流れ

1. **アプリを開き「llms.txt を用意する」ボタン（または初回案内）を押す**
   - 例: 「llms.txt 用のファイルを自動作成」ボタン。

2. **アプリが Shopify の「コンテンツ → ファイル」に 1 個の .txt を作成する**
   - 中身は「ひな形」（空、または「ここにサイトの意図を書いてください」などの短い説明文）。
   - 作成後、**Files API が返す CDN URL** をアプリが受け取る。

3. **アプリがその URL を保存する**
   - 案 A と同じく、**アプリ所有メタフィールド**（例: `llmo.llms_txt_url`）に保存。

4. **ユーザーは「コンテンツ → ファイル」で、そのファイルを開いて中身を編集する**
   - ファイルはすでに 1 個あるので、それを編集するだけ。URL は変わらない（同じ CDN URL のまま中身だけ更新される）。

5. **テーマで「LLMO head」ブロックを有効にすると、head にその URL が link で出る**
   - 案 A と同じ。Liquid は `app.metafields.llmo.llms_txt_url` を読むだけ。

### 技術側のやること（イメージ）

| 役割 | やること |
|------|----------|
| **アプリ（管理画面）** | 「llms.txt を自動作成」のようなボタンを用意。押したら **Files API（GraphQL）** で `stagedUploadsCreate` → ひな形テキストをアップロード → `fileCreate` でファイル作成。返ってきた **file.url**（CDN URL）をメタフィールド `llmo.llms_txt_url` に保存。まだ URL が無いときは「作成する？」と案内、既にあるときは「再作成」か「そのまま使う」を選べるようにするなど。 |
| **Theme Extension（Liquid）** | 案 A と同じ。`app.metafields.llmo.llms_txt_url` があればその URL を &lt;link&gt; に使う。 |
| **データの流れ** | ボタン → アプリが Files API で 1 ファイル作成 → URL 取得 → メタフィールドに保存 → ブロックがその URL を参照 → head に 1 本の link。 |

**イメージ図（B 案）**

```
[ユーザー] アプリで「llms.txt を用意」ボタンを押す
       ↓
[アプリ] Files API でひな形 .txt を 1 個作成 → CDN URL を取得
       ↓
[アプリ] メタフィールド（llmo.llms_txt_url）にその URL を保存
       ↓
[ユーザー] 管理画面「コンテンツ→ファイル」でそのファイルを開き、本文を編集
       ↓
[テーマブロック] app.metafields.llmo.llms_txt_url を読む
       ↓
[head] <link href="(そのCDN URL)" ... />
```

- **メリット**: ユーザーは「URL をコピーして貼る」をしなくてよい。最初の 1 クリックで「置き場所」ができ、あとは中身を書くだけ。
- **デメリット**: アプリ側で Files API（staged upload + fileCreate）の実装が必要。ひな形の文言やファイル名のルールは **docs/llms-txt_雛形の方針.md** で方針を整理している（サイト種別で雛形を分ける案）。

---

## 両方やる場合（推奨イメージ）

- **基本は B**：多くのユーザーは「ボタン 1 回 → ファイルができる → 中身を編集」で完結。
- **上級用に A も用意**：「llms.txt の URL」を直接入力できる欄を 1 つ用意し、  
  - 入力があれば **その URL を優先**してメタフィールドに保存（B で作った URL を上書きできる）、  
  - なければ B で作った URL を使う、  
  という形にすると、「既存のファイルの URL」や「外部ホストの URL」も使える。

---

## まとめ（一言で）

| 案 | 一言で |
|----|--------|
| **A** | ユーザーが「llms.txt の URL」をアプリの設定欄に **貼り付けて保存**。ブロックはその URL を &lt;link&gt; に使う。 |
| **B** | ユーザーがアプリで **「作成」ボタンを押す**と、アプリが Shopify の Files にひな形の .txt を 1 個作り、その **CDN URL を自動で保存**。ブロックはその URL を &lt;link&gt; に使う。ユーザーは「コンテンツ→ファイル」で中身を編集するだけ。 |

どちらでも、**Theme Extension が参照するのは「1 本の URL（メタフィールド）」** で共通。違いは「その URL を誰が・どう用意するか」だけ。
